## 바이브코딩으로 블로그를 만들었다

Claude에게 말했다.

> "Next.js + TypeScript + Tailwind로 개인 기술 블로그 만들어줘. 마크다운 파일로 포스트 관리하는 구조로."

잠깐 기다렸더니 다음이 갖춰져 있었다:

- `src/lib/posts.ts` — `gray-matter`로 frontmatter 파싱, `remark`로 HTML 변환
- `src/types/post.ts` — `Post`, `PostMeta` 타입
- 홈 / 블로그 목록 / 포스트 상세 / About 페이지
- 다크모드 대응 레이아웃

`npm run build` 를 돌렸더니 오류 없이 통과했다. 평소라면 보일러플레이트 세팅, 디렉토리 구조 고민, 라이브러리 찾기에 반나절은 썼을 것이다.

바이브코딩은 개발 속도를 높여주는 도구다. 판단과 책임은 여전히 사람 몫이지만, 지루한 반복 작업에서 해방된 시간을 뭘 만들지 생각하는 데 쓸 수 있다.

---

## 이전에 만든 것들

이 블로그 전에도 몇 가지를 만들어봤다.

첫 번째는 순수 HTML/CSS로 만든 자기소개 페이지였다. 코딩이라고 부르기도 민망한 수준이었지만, 직접 만든 페이지가 브라우저에 뜨는 걸 보는 게 신기했다.

그 다음은 React + Vite로 만든 한국 날씨 앱. 지도에서 행정구역을 클릭하면 실시간 날씨를 보여주는 구조였다. API 연결하고, 지도 컴포넌트 붙이고 — 처음으로 뭔가 동작하는 걸 만들었다는 느낌이 있었다.

그리고 이 블로그다.

---

## 로컬에서 브라우저까지

블로그 포스트는 `posts/` 폴더에 마크다운 파일로 관리한다. 파일 이름이 URL slug가 된다.

`npm run dev`를 켜두면 파일 저장하자마자 브라우저에서 바로 확인할 수 있다.

![로컬 개발 서버에서 확인하는 블로그 화면](/images/posts/blog-local.png)

로컬에서 확인이 되면 GitHub에 올린다.

```bash
git add .
git commit -m "post: 새 글 추가"
git push origin main
```

![GitHub에 푸쉬된 커밋 화면](/images/posts/blog-github.png)

GitHub 레포를 Vercel에 연결해두면, `main` 브랜치에 푸쉬할 때마다 자동으로 빌드하고 배포한다. 처음 연결할 때만 Vercel 대시보드에서 레포를 가져오면 된다. 이후엔 푸쉬만 하면 끝이다.

![Vercel 배포 완료 화면](/images/posts/blog-vercel.png)

글 하나 올리는 흐름이 이렇다:

1. **로컬에서 `.md` 작성** — `posts/` 폴더에 파일 추가
2. **`npm run dev`로 확인** — 저장 즉시 미리보기
3. **`git push origin main`** — GitHub에 커밋 푸쉬
4. **Vercel 자동 배포** — 푸쉬 감지 → 빌드 → 배포
5. **브라우저에서 접속** — 발급된 URL로 누구나 접근 가능

## 처음으로 레포를 직접 만들었다

회사에서 GitHub을 아예 안 써본 건 아니다. 콘텐츠 작업을 하면서 파일 수정하고, 커밋하고, PR 올리는 일을 해왔다. 근데 항상 누군가 이미 만들어놓은 레포에 들어가서 하는 일이었다.

직접 레포를 만들고, 코드를 올리고, 배포까지 연결하는 건 이번이 처음이었다.

하면서 자연스럽게 PR이 왜 존재하는지도 이해가 됐다. 회사에서 PR을 올릴 때는 "원래 이렇게 하는 거구나" 하고 따라했는데 — 혼자 레포를 관리해보니 그 흐름이 왜 생겼는지가 보였다.

여러 명이 같은 레포에서 작업하면 서로의 코드가 충돌하거나, 버그가 검토 없이 배포될 수 있다. PR은 **"내 변경을 main에 합치기 전에 한 번 봐줘"** 라는 요청이다. 코드든 글이든 검토 후 반영이라는 흐름은 같다.

지금 이 블로그는 혼자 관리하니까 main에 직접 푸쉬해도 충분하다. 글이 쌓이고 뭔가 협업할 일이 생기면 그때 브랜치 방식으로 전환하면 된다.

## 아직 완전히 이해된 건 아닌데

Vercel이 빌드를 어떻게 하는지, CI/CD가 정확히 어떻게 돌아가는지 — 아직 겉핥기 수준이다.

그래도 직접 해보기 전까지 그냥 절차로만 알고 있던 것들이 만들면서 하나씩 이해되는 중이다. 이해도보다 경험이 먼저인 것 같다.
